# py-lab — учебный проект

Лабораторная работа на Python: система задач с разными источниками данных и единым процессором обработки

## О чём проект

Небольшая система управления задачами, которая показывает:

- **Модели данных** — задача (`Task`) с приоритетом, статусом, сериализацией в JSON
- **Контракты (Protocol)** — единый интерфейс для источников задач без наследования
- **Источники задач** — файл (JSON), API (заглушка), генератор
- **Процессор** — один процессор обрабатывает задачи из любого источника по одному контракту
- **Дескрипторы и property** — контроль доступа к атрибутам, валидация, вычисляемые свойства
- **Логирование и тесты** — pytest, покрытие кода, проверка контрактов

Что сделано: dataclasses, typing (Protocol, runtime_checkable), декораторы, data/non-data дескрипторы, специализированные исключения, структурированное логирование

## Лабораторная работа №1. Источники задач и контракты (duck typing)

**Цель:**  
Освоить duck typing и контрактное программирование на примере источников задач.

**Постановка задачи:**  
Реализовать подсистему приёма задач в платформе обработки задач. Задачи могут поступать из различных источников, не связанных наследованием, но обязанных реализовывать единый поведенческий контракт.

**Примеры источников задач:**
- задачи, загружаемые из файла;
- задачи, генерируемые программно (генератор);
- задачи, получаемые из API-заглушки.

**Функциональные требования:**
- все источники задач должны реализовывать единый контракт (метод `get_tasks()`);
- запрещено использование общего базового класса для источников;
- контракт должен быть описан с помощью `typing.Protocol`;
- должна быть реализована как минимум одна runtime-проверка соблюдения контракта;
- архитектура должна позволять добавление новых источников без изменения существующего кода.

**Технические требования:**
- использование `Protocol` и `runtime_checkable`;
- осмысленное применение `isinstance()` и/или `issubclass()`;
- аннотации типов для всех публичных интерфейсов.

**Что реализовано:**

- В `src/contracts.py` определён протокол `TaskSource` с методом `get_tasks() -> Iterable[Task]` и декоратором `@runtime_checkable`.
- Реализованы три источника (без общего базового класса):
  - `FileTaskSource` – читает задачи из JSON-файла.
  - `ApiTaskSource` – возвращает тестовый набор задач.
  - `GeneratorTaskSource` – генерирует указанное количество задач с циклическим приоритетом.
- Написана функция `is_task_source()` и `assert_is_task_source()` для runtime-проверки контракта.
- В `src/processor.py` класс `TaskProcessor` принимает любой источник, соответствующий `TaskSource`, и обрабатывает задачи. Декораторы `@validate_source` и `@log_processing` демонстрируют расширяемость.
- Добавлены тесты: `test_contracts_and_logging.py`, `test_sources.py`, `test_processor.py`.

**Ожидаемый результат:**  
Модуль приёма задач, способный работать с несколькими источниками через единый контракт.

## Лабораторная работа №2. Модель задачи: дескрипторы и @property

**Цель:**  
Освоить управление доступом к атрибутам и защиту инвариантов доменной модели через пользовательские дескрипторы и `property`.

**Постановка задачи:**  
Реализовать модель задачи (`Task`) в рамках платформы обработки задач с корректной инкапсуляцией и валидацией состояния.

**Атрибуты задачи:**
- идентификатор задачи (read-only, валидация);
- описание (строка, непустое);
- приоритет (перечисление, валидация);
- статус (перечисление, валидация);
- время создания (datetime);
- вычисляемые свойства (например, возраст, срочность).

**Функциональные требования:**
- использование пользовательских дескрипторов для валидации атрибутов;
- использование `property` для вычисляемых или защищённых свойств;
- предотвращение некорректных состояний объекта;
- генерация специализированных исключений при нарушении инвариантов.

**Технические требования:**
- разделение публичного API и внутреннего состояния;
- демонстрация различий между data и non-data дескрипторами;
- аннотации типов и документация.

**Что реализовано:**

### Дескрипторы (`src/descriptors.py`)

- **`ReadOnlyIdDescriptor`** – data-дескриптор (есть `__set__`).  
  - Установить ID можно только один раз (при создании).  
  - При попытке изменения – `ImmutableAttributeError`.  
  - Валидация типа (int) и значения (≥0).  
  - Запрет удаления.

- **`ValidatedPriorityDescriptor`** – data-дескриптор.  
  - Принимает `int` или `TaskPriority`.  
  - Автоматически конвертирует int в перечисление.  
  - Некорректные значения → `InvalidPriorityError`.

- **`ValidatedStatusDescriptor`** – data-дескриптор, аналогично для статуса (`InvalidStatusError`).

- **`NonDataDescriptionDescriptor`** – **non-data дескриптор** (нет `__set__`).  
  - Показывает разницу в приоритете поиска атрибутов: если в `__dict__` экземпляра есть ключ `description`, возвращается он; иначе – значение из хранилища `_description`.  
  - При установке `task.description = ...` создаётся запись в `__dict__`, переопределяющая дескриптор.  
  - Это демонстрирует поведение non-data дескриптора.

- **`ValidatedStringDescriptor`** – обобщённый data-дескриптор для валидации строк (длина, пустота). Показывает переиспользование логики.

### Свойства (`@property`) в `Task`

- `is_completed`, `is_pending`, `is_in_progress` – проверки статуса.
- `age_seconds` – возраст задачи в секундах (от `created_at` до текущего момента).
- `age_formatted` – человеко-читаемое представление возраста (сек/мин/ч).
- `is_urgent` – `True` для приоритетов `HIGH` или `CRITICAL`.

### Исключения (`src/exceptions.py`)

- `TaskInvariantError` – базовое для всех нарушений инвариантов задачи.
- `InvalidTaskIdError`, `InvalidPriorityError`, `InvalidStatusError`, `EmptyDescriptionError`, `ImmutableAttributeError`.

### Класс `Task`

- Использует дескрипторы для `id`, `priority`, `status`, `description`.
- В `__init__` выполняется валидация описания, затем установка через дескрипторы с обработкой ошибок.
- Методы `to_dict()`, `to_json()`, `from_dict()`, `from_json()` для сериализации.
- Переопределены `__str__`, `__repr__`, `__hash__`, `__eq__` (по `id`).
- Метод `update_description()` показывает, как можно работать с non-data дескриптором (создавать или удалять атрибут экземпляра).

### Коллекция `TaskCollection`

- Хранит задачи в списке и словаре для быстрого доступа по `id`.
- Методы: `add()`, `remove()`, `get()`, `get_all()`, `filter()`.
- Поддержка `len()`, `in`, итерации, доступа по квадратным скобкам (`__getitem__`).

### Тесты (`tests/test_descriptors.py`, `tests/test_models.py`)

- Проверка работы каждого дескриптора (корректные и некорректные значения, неизменяемость ID).
- Демонстрация различий data и non-data дескрипторов (переопределение `description` через `__dict__`).
- Тесты вычисляемых свойств, сериализации, коллекции.
- Покрытие тестами >85%.

**Ожидаемый результат:**  
Класс `Task` с чистым и безопасным публичным API, демонстрирующий управление атрибутами через дескрипторы и `property`.
